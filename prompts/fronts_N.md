# Examine N for fronts

## Goals

Explore the density profiles at fronts in a single tile of the LLC4320 grid.  Also consider calculations of the bouyancy frequency.

## Functionality

- The user will input a 3D density field file, which was generated by the generate_tile_density script in llc4320-native-grid-preprocessing.
- The user will provide the gradb2 field for the full globe
- The user will provide a labeled fronts file which contains the fronts labeled as integers.
- The user will provide a index fronts file which contains the box in i,j (given as x0, y0, x1, y1) for each front.  It also contains a label which is the unique identifier for the front.
- The code will find the N (an integer default to 10) strongest fronts in the tile based on the gradb2_p90 property of the fronts.
- The code will find the i,j coordinates for the peak of gradb2 for each of the N fronts.
- The code will generate a plot of the density profile for each of the N fronts versus depth.  This will be a 1 panel figure with a different color per front.  The legend will be the front label.

## Planning

### Plan

#### Data the script consumes

1. **Density tile NetCDF** — produced by `dev/pot_density/generate_tile_density.py` in
   `llc4320-native-grid-preprocessing`. Contents:
   - `sigma0(k, j, i)` — potential density (kg m^-3), shape `(51, 720, 720)`.
   - `XC(j, i), YC(j, i)` — 2D lon/lat for the tile.
   - `Z(k)` — depth (negative downward, length 51).
   - Attrs we will rely on: `tile_index`, `face_index`, `rect_i_start`, `rect_j_start`,
     `timestamp`.
   - **Coordinate gotcha**: `(j, i)` here are *face-local* axes, possibly rotated relative
     to the global rect grid (handled by `dev/pot_density/tile_mapping.py`).
2. **Global gradb2 field** — float32 2-D array on the rect grid, shape `(12960, 17280)`.
   File format expected: `.npy` or `.npz` (CLI accepts either).
3. **Global labeled-fronts** — int32 array, same shape as gradb2; `0` is background, positive
   ints are unique front labels (one per detected front). Format: `.npy`.
4. **Global front index** — parquet/csv produced by
   `fronts.properties.io.write_front_index` with columns
   `label, name, x0, y0, x1, y1` plus enriched columns (incl. `gradb2_p90`) from the
   colocate step.

#### High-level steps

1. **CLI** (`argparse`):
   `--density-tile`, `--gradb2`, `--labels`, `--front-index`, `--front-properties`
   (separate parquet with `gradb2_p90`, joined on `label`), `--N` (default 10),
   `--top-fronts-csv` (optional; if supplied, skip re-finding step), `--outdir`
   (default `.`). All output files are written to `--outdir` (CWD by default).
2. **Load** the density tile with `xr.open_dataset`. Pull `rect_i_start`, `rect_j_start`,
   `face_index`, and `timestamp` from attrs. Build the rect-grid tile slices
   `rect_j_slice = slice(rect_j_start, rect_j_start + 720)` and the analogous
   `rect_i_slice`. Form the output stem
   `density_profiles_tile{tile_index:03d}_{YYYYMMDDTHH}_topN{N}`.
3. **Resolve the top-N peaks** — *short-circuit branch*:
   - If `--top-fronts-csv` is supplied and exists, OR if a CSV with the default name
     already exists in `--outdir`, load it and skip steps 4–7. The CSV columns are
     `label, name, gradb2_p90, i_rect, j_rect, i_face, j_face, lon, lat` (see
     Modification 1 + Follow-up Clarification 1). Validate `len(csv) == N`; if the
     stored `N` differs, error out and tell the user to delete/rename the stale file.
4. **Identify in-tile fronts** *(only if no CSV)*:
   - `front_index = load_front_index(--front-index)` → columns `label, name, x0, y0, x1, y1`.
   - `front_props  = pd.read_parquet(--front-properties)` → must contain `label` and
     `gradb2_p90`.
   - Inner-join the two on `label`. Drop rows missing `gradb2_p90`.
   - Tile-overlap filter (bbox in rect grid; option (i) — partial overlap counted):
     `x0 < i1_rect and x1 >= i0_rect and y0 < j1_rect and y1 >= j0_rect`.
5. **Pick top-N candidates**: sort the filtered table by `gradb2_p90` descending. Keep
   the **full sorted list** as the candidate pool (not just the top N) — step 7 walks
   it in order and promotes the next candidate whenever a front turns out to have zero
   in-tile pixels (Round-2 Clarification 5). The final accepted set is at most `N`.
6. **Build the rect→face-local lookup** using
   `tile_mapping._get_lookup_arrays()`. Restrict the three maps to
   `[rect_j_slice, rect_i_slice]` → three `(720, 720)` int arrays
   (`face_id_tile`, `j_face_tile`, `i_face_tile`). Sanity-check
   `np.unique(face_id_tile) == [face_index]`.
7. **Find the peak-gradb2 pixel for each front** (per Round-2 Clarification 5: keep
   bbox-overlapping fronts; only compute the peak over the in-tile portion):
   - `gradb2 = np.load(--gradb2, mmap_mode='r')`; `labels = np.load(--labels, mmap_mode='r')`.
   - `gradb2_tile = gradb2[rect_j_slice, rect_i_slice]`;
     `labels_tile = labels[rect_j_slice, rect_i_slice]`.
   - Walk the bbox-overlapping table in `gradb2_p90`-descending order. For each label:
     - `mask = (labels_tile == label)`.
     - If `mask.sum() > 0` → `argmax` over `np.where(mask, gradb2_tile, -inf)` →
       tile-local `(j_local, i_local)`; accept this front.
     - If `mask.sum() == 0` (bbox overlaps but the label has zero pixels inside the
       tile — rare edge): warn, **skip this front, and promote the next-strongest
       bbox-overlapping front** so we still aim for `N` accepted fronts.
     - Stop when N fronts are accepted or the candidate list is exhausted (warn if
       short).
     - Rect coords: `i_rect = i0_rect + i_local`, `j_rect = j0_rect + j_local`.
     - Face-local coords: `j_face = j_face_tile[j_local, i_local]`,
       `i_face = i_face_tile[j_local, i_local]`.
     - `lon = XC[j_face, i_face]`, `lat = YC[j_face, i_face]`.
8. **Write the top-N CSV** (Modification 1): one row per resolved front, columns from
   step 3 above, ordered by `gradb2_p90` descending. Written to
   `{outdir}/{stem}.csv`.
9. **Extract density profiles**:
   `profiles[n, :] = sigma0.values[:, j_face, i_face]` (a small fancy-index gather).
10. **Plot 1 — density profiles** (single panel, `matplotlib`):
    - x-axis: `sigma0` (kg m^-3); y-axis: depth `Z` (m), inverted so surface is on top.
    - One line per front; colour from a fixed cycler: `tab10` for N≤10 else `viridis`
      sampled at `N` points. Cache `colors[n]` so plots 1 and 2 stay in sync.
    - Legend entries: `name` (per Clarification 2 — note these strings are long;
      legend `fontsize='x-small'`, placed `bbox_to_anchor=(1.02, 1)` outside the
      axes).
    - Title: `Tile {tile_index}  {timestamp}\nTop-{N} fronts by gradb2_p90`.
    - Saved as `{outdir}/{stem}.png`.
11. **Plot 2 — log10(gradb2) overlay** (Modification 2):
    - `imshow(np.log10(np.maximum(gradb2_tile, eps)), origin='lower')` on rect-grid
      tile-local axes (0..720). Using rect-grid here keeps the dots from step 7
      consistent without an extra coordinate transform — see Follow-up Clarification 2.
    - Overlay `ax.scatter(i_local, j_local, c=colors, s=40, edgecolor='white')`,
      one dot per front, colours matching Plot 1.
    - Colourbar with label `log10(gradb2)`.
    - Title: `Tile {tile_index}  {timestamp}\nlog10(gradb2) with top-{N} peaks`.
    - Saved as `{outdir}/{stem}_gradb2map.png`.

### Clarifications

1. **Which parquet has `gradb2_p90`?** `write_front_index` only stores
   `label, name, x0..y1`. The `gradb2_p90` column lives in the *colocated front
   properties* parquet (see `ColocateFrontProperties_Global_Viz.ipynb`). Should the
   CLI take **one** parquet path (the enriched one, which can also serve as the
   bbox source) or **two** (index + properties, joined on `label`)? Plan currently
   assumes one enriched parquet.

The code should use both, joined on label.

2. **Legend label**: `label` (small int) or `name` (TIME_LAT_LON string)? `name` is
   informative but long; with N=10 the legend will get crowded. Default in the plan is
   `label`.

The code should use name.

3. **Buoyancy frequency**: the *Goals* mention it but *Functionality* specifies a
   single-panel figure. Should we (a) skip N² for this prompt, (b) add a 2nd panel, or
   (c) emit a separate `*_N2.png`?

We are ignoring buoyancy frequency for now.

4. **Tile-edge fronts**: a front whose bbox overlaps the tile may have its `gradb2`
   peak *outside* the tile. Two options: (i) include such fronts and compute the peak
   only over the in-tile portion (current plan), or (ii) restrict to fronts whose bbox
   lies entirely inside the tile. Plan is (i) because (ii) discards most fronts.

Use option (i). 

5. **Output path & filename pattern**: default
   `density_profiles_tile{tile_index:03d}_{YYYYMMDDTHH}_topN{N}.png` next to the
   density-tile NetCDF — confirm or specify a directory.

Use the current directory.
